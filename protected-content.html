Internship schedule

  • Phase 1
    Research Active Directory security
    Phase 1 was a preparation phase. Preparation is essential because a good knowledge of active directory security is required before the Silverfort insights can be understood properly, good security advice can be given and Silverfort policies can be created.
    Documentation
    Fase 1: Security voor AD
  • Phase 2
    Installation
    In phase 2, the goal was to install Silverfort for the child domain. This was done by first looking at the installation videos from Silverfort academy. Next, I made a schematic representation, filled in the IS4U Silverfort playbook and wrote an installation manual.
    Documentation
    Fase 2: Silverfort installatie proces
  • Phase 3
    Security audit with and without Silverfort
    In phase 3, I was tasked with investigating the security of a child domain sil.is4ulab.be.
    Part 1: find configuration mistakes with opensource tools
    Part 2: use Silverfort to find configuration mistakes.
    The purpose of the assignment was to discover why Silverfort can add value to active directory security.
    Documentation
    Fase 3: Security onderzoek
    Fase 3: Security onderzoek met Silverfort
  • Phase 4
    Attack and defense
    During the period from 23 March to 29 March, I carried out several attacks in the child domain sil.is4ulab.be. The purpose of this assignment is to demonstrate how far an attacker can get through misconfigurations within a domain.

    Next, I took a look at the logs during the attack. From these logs, I had to find the specific logs when attacks occurred. From the logging info, I had to create Silverfort policies to prevent attacks like this in the future.

    Documentation
    Fase 4: Aanvalsmethode
    Fase 4: Observaties Silverfort
    Fase 4: Policy test

  • Additional phase
    Integration with SentinelOne
    In this additional phase, I tried to perform an integration of Silverfort and SentinelOne. This way, there is not only identity-based security but also endpoint-based security. The goal is that in Silverfort, SentinelOne's observations come in and affect the risk factor of users/resources. Also, Silverfort will have to influence SentinelOne: if a user logs in multiple times but gets denied at MFA, this user should be logged off from all endpoints.
